## 2. 上游同步与更新指南

当原作者（上游）更新了代码后，你可以放心大胆地在 GitHub 页面点击 `Sync fork`。因为上述四个文件都是全新的，**绝不会产生合并冲突**。

但同步之后，你可能希望把官方在 `GENERAL.txt` 或 `WRT-CORE.yml` 中的最新改进吸收进你的专属配置中，以下是检查和同步策略：

### 📌 `MY-IPQ60XX.txt` 同步策略
**基本无需同步**。正如你所说，官方极少去改变已有设备的 `CONFIG_TARGET_...=y` 格式，你可以永久保持当前内容不变，无需理会官方的变动。

### 📌 `MY-WRT-CORE.yml` 同步策略
核心工作流如果官方有大改（例如更换了编译系统、更新了环境脚本等），你可以通过对比检查。
- **检查方法**：利用对比工具查看官方 `WRT-CORE.yml` 的近期提交记录，或者直接对比 `WRT-CORE.yml` 和 `MY-WRT-CORE.yml`。
- **更新方式**：如果官方有重大更新，建议直接全选复制最新的 `WRT-CORE.yml` 覆盖到你的 `MY-WRT-CORE.yml` 中，然后**只需把里面 172 行左右的 `GENERAL.txt` 重新改回 `MY-GENERAL.txt` 即可**。

### 📌 `MY-GENERAL.txt` 同步策略（重点！）
**你的需求**：官方新增的想保留，官方调整的想跟随，但自己主动删减的内容不能被官方重新加回来。

为了满足这个需求，**最轻松、且最符合 OpenWrt 编译机制的做法是：利用“末尾覆盖原则”，而不要直接去删减中间的代码！**

OpenWrt 的 `.config` 配置文件合并时，**写在文件越后面的配置，优先级越高**。
因此，建议你把 `MY-GENERAL.txt` 分为上下两部分来维护：

```text
# ==========================================
# 第一部分：官方最新配置 (直接复制，保持原样)
# ==========================================
CONFIG_PACKAGE_luci-app-homeproxy=y
...
(此处全选复制官方 GENERAL.txt 的最新内容)
...

# ==========================================
# 第二部分：我的自定义覆盖配置 (永远放在最底部)
# ==========================================
# 需求1：删除不想要的官方插件（改为 =n 或者不设定）
CONFIG_PACKAGE_luci-app-upnp=n
# CONFIG_PACKAGE_luci-app-samba4 is not set

# 需求2：强制开启我想加的插件
CONFIG_PACKAGE_luci-app-mycustom=y
```

**按照这种结构，未来的同步步骤将变得极其简单：**
1. 官方更新后，你直接打开官方最新的 `GENERAL.txt`，`Ctrl+A` 全选，`Ctrl+C` 复制。
2. 打开你的 `MY-GENERAL.txt`，把**第一部分**全选覆盖掉（第二部分保持不动）。
3. **完成！**

**原理解释**：官方新增的内容自然会带入第一部分；官方调整的参数也会更新在第一部分；而那些你“故意删除”的内容，因为你在第二部分写了 `=n`，根据末尾覆盖原则，官方在第一部分怎么 `y` 都会被你第二部分的 `n` 强制覆盖掉（相当于不添加）。这样就完美实现了你的所有要求，而且根本不需要写任何复杂的自动比对脚本。
